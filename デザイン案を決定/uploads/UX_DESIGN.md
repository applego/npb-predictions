## 1. ページ構成の最適化

### 1-1. 現状の問題分析

| 問題 | 現行URL | 影響 |
|------|---------|------|
| `/standings` と `/rankings/commentators` で「誰が何点か」の重複 | 2ページ | ユーザーが迷う |
| `/seo/past-seasons` が独立ページとして存在 | 1ページ | 各ページに年セレクタがあるので不要 |
| タイトル予想結果の専用ページなし | — | タイトル的中がスコアに見えない |
| マイページなし | — | 自分の予想を素早く確認できない |
| ナビ項目が多い（HOME/PREDICTIONS/RANKINGS/GROUPS/NEWS/STANDINGS/ARCHIVE） | 7項目 | モバイルで溢れる |

### 1-2. 統合・整理方針

#### /standings の廃止と /league への統合

現行の `/standings`（友人5人のスコアボード）は `/league` に改称し、
`/rankings/commentators`（156人解説者ランキング）とは明確に分離する。

- `/league` ... 友人5人のスコアボード（内輪リーグ）
- `/rankings` ... 156人の解説者的中率ランキング（外向け情報）

この2つは「規模」と「目的」が全く異なるため、統合ではなく**命名の明確化**で解決する。

#### /seo/past-seasons の廃止

各ページに年セレクタが存在するため不要。SEO用途は `/seo/[year]` 配下で継続。

#### /rankings 直下の整理

```
/rankings              → /rankings/commentators にリダイレクト（現行維持）
/rankings/commentators → 解説者ランキング（156人）
/rankings/all-time     → 通算ランキング
```

### 1-3. 提案ページ構成

```
/                           HOME（ヒーロー + クイックアクセス + マトリクスプレビュー + ニュース）
/league                     リーグスコアボード（旧 /standings）
/predictions                予想マトリクス比較（セ・パタブ切替）
/predictions/new            予想登録フォーム
/titles                     タイトル予想ページ（新規）
/me                         マイページ（新規、ログイン必須）
/rankings/commentators      解説者ランキング（156人）
/rankings/all-time          通算ランキング
/users/[userId]             個人予想詳細
/groups                     対決グループ一覧・作成
/groups/[slug]              グループ詳細・グループ内ランキング
/groups/join                招待コードで参加
/news                       ニュースフィード
/settings                   テーマ・フォント設定
/admin                      管理画面
/seo/[year]                 SEO静的ページ（検索流入用、内部リンク非表示）
```

廃止対象:
- `/standings` → `/league` に301リダイレクト
- `/seo/past-seasons` → `/league` に301リダイレクト
- `/archive/[year]` → `/league?year=[year]` に301リダイレクト

### 1-4. ナビゲーション整理

#### デスクトップ（7→5項目）

```
NPB PREDICTIONS  |  リーグ  予想  タイトル  ランキング  グループ  |  ニュース  [ログイン]
```

- 削除: STANDINGS（→ リーグに改称・統合）、ARCHIVE（廃止）
- 追加: タイトル予想（新規）
- NEWS は補助的ナビ（右端）に移動してメインナビを軽量化

#### モバイルハンバーガーメニュー（既存実装あり）

```
[NPB PREDICTIONS]  [≡]

展開後:
─────────────────
 リーグ
 予想を見る
 タイトル予想
 ランキング
 グループ
─────────────────
 ニュース
 設定
─────────────────
 [ログイン / マイページ]
```

### 1-5. URL 設計原則

- 英語URL（既存方針維持） + 日本語ラベル（ナビ表示）
- 年パラメータは `?year=YYYY`（既存方針維持）
- スラッグはすべて kebab-case

## 4. 推移表示の設計

### 4-1. 現状

`/users/[userId]` に `ScoreHistoryChart` コンポーネント（既存）が存在する。
ただし**年次グラフ**（シーズン単位の棒/折れ線）のみで、
シーズン内の月次推移（「この時期は1位だった」）は未実装。

### 4-2. 2種類の推移表示

#### Type A: シーズン間推移（既存）— 年次スコア推移

```

        │               ●
    +20 │          ●
        │     ●
    +10 │
     0  ●─────────────────────────

```

用途: マイページ・個人詳細ページ
データ: `getUserHistory()` → `scoreSnapshots`（各シーズン最新スナップショット）
状態: 実装済み（`ScoreHistoryChart.tsx`）

#### Type B: シーズン内推移（新規）— 月次順位推移


## 9. 設計判断メモ

### なぜ /standings と /rankings/commentators を統合しない？

友人5人のスコアボード（`/league`）は「身内で競う」内輪コンテンツ。
解説者ランキング（`/rankings/commentators`）は「156人の公式解説者を比較する」外向けコンテンツ。
ターゲット・データ量・更新頻度がすべて異なるため、統合よりも**命名の明確化**が適切。

### なぜ /me を新設し /users/[userId] を残す？

`/users/[userId]` は他のユーザーが自分のプロフィールを見る用途で維持する。
`/me` はログインユーザー専用でグループ情報や編集ボタンを追加するため別ページとする。
実装は `/users/[userId]` のコンポーネントを最大限再利用してコードの重複を最小化。

### タイトル予想の `actual_title_winners` テーブルについて

現行 DB には `titlePicks`（予想側）はあるが「実際の結果」を格納するテーブルがない。
Admin画面からシーズン終了後に入力 → 自動的に的中判定を計算する設計が適切。
Admin UI への追加は Phase 3 で対応。

### /seo/* ページの扱い

検索流入用 SEO ページ（`/seo/[year]/[league]/final-standings` 等）はナビから外して維持。
`robots.txt` / `sitemap.xml` への掲載は継続。ユーザーが意図的にナビから辿る導線は不要。
